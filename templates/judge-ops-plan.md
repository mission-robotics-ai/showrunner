# Judge Coordination Plan

**How to use:** Internal ops doc, not sent to judges — it's the source the judge-comms.md blocks pull from. Fill in the panel, the timeline, and the rubric, then get the rubric approved by the event host before Block B of judge-comms.md goes out. The "Open decisions" section at the bottom is meant to be worked through, not skipped — every unresolved row there blocks something downstream.

## What we owe judges, and when

### Now / pre-event (T-2)
Confirm the ask in writing — the date, the time window, the address, what to expect on the day, food, and parking. This is Block A in `judge-comms.md`. The goal is a firm yes with logistics locked, so nobody is figuring out where to park at the last minute.

### T-1 (day or two before)
Send the rubric, how scoring works, and the submission list/format so judges can pre-read if they want to. This is Block B in `judge-comms.md`. Whether pre-reading is expected or optional is a host decision — default the copy to optional unless told otherwise.

### Day-of morning
Check-in with a named greeter, a printed scoring sheet per judge, and a short verbal briefing before demos. This is Block C in `judge-comms.md` for the morning text, plus the physical scoring sheets printed the night before.

## PROPOSED rubric — host approves or edits before it ships

[N] criteria, each scored 1–5:

1. **[CRITERION THAT DEFINES YOUR EVENT'S IDENTITY]** — [the thing that most directly reflects what the event is actually testing for]. This is the core of the whole event.
2. **[TECHNICAL DIFFICULTY]** — how hard was what they attempted?
3. **[INGENUITY]** — cleverness and novelty of the approach.
4. **[DEMO QUALITY]** — clarity of the demo. Did they show it working and explain what we were seeing?

**Aggregate:** sum the criteria per judge (max [5×N] per judge), then sum across judges. Highest total wins.

**Weighting option:** if the host wants the event's defining criterion to bite harder, weight that criterion double. Optional, host's call — but recommended whenever one criterion is the actual point of the event (e.g., "does it work on real hardware" vs. a slick pitch).

**Tie-breaks:** the team with the higher score on the defining criterion wins the tie. If still tied, the judges confer and rule.

## Logistics judges need answered

- **In person or virtual** — say so explicitly in Block A; don't leave it ambiguous.
- **Time commitment** — framed up front so nobody books something else during the window.
- **Food provided** — name who owns it; judges eat with the room.
- **Where to sit** — a judges' area with sightlines to the demo space, shown at briefing.
- **Who runs the room** — name the host; name the on-site partner for anything operational.
- **Tie-breaks** — resolved by the rubric above, then judges confer.
- **Sub-panel awards** — open question: does any judge join for a single sponsor-specific award only, or judge the whole panel? Needs a ruling so the briefing is clean.

## Prize tracks (context for judges)

List sponsor-funded prizes here so judges know the stakes going in — see `prize-fulfillment-sheet.md` for the tracker that pays these out. Note whether these are separate track winners or one grand prize plus track callouts; the rubric works either way, it just changes how totals get read.

## Open decisions flagged for host sign-off

1. **Rubric sign-off** — approve, edit, or replace the proposed rubric above. Everything downstream (scoring sheets, Block B) waits on this.
2. **Grand prize vs per-track** — one overall winner, or a winner per sponsor track? Changes how winners are announced.
3. **Sub-panel structure** — does any judge join for a single award only, or the full panel? Affects the briefing and the deliberation.
4. **Pre-read vs live-only** — do judges pre-read submissions ahead of time, or judge live-only on the day? Confirm the default framing in Block B.
